# Sprite atlas tearing: grid math vs. authoritative per-frame rects

**Date:** 2026-05-23
**Tags:** `bevy-asset`, `texture-atlas`, `manifest-vs-derived`, `silent-filter`, `inter-frame-padding`
**Prompt level:** B (post-error: "kernel guide sprite is tearing")

---

## Background

Ambition's character spritesheets are generated by a Python toolchain
that emits three sidecar files per target: `<name>_spritesheet.png`,
`<name>_spritesheet.yaml` (human-readable), and
`<name>_spritesheet.ron` (machine-readable, consumed at runtime by
`SheetRegistry`). Each RON row lists the row's animation, frame
count, duration, AUTHORED `row_index`, AND a `rects: Vec<FrameRect>`
giving the exact pixel x/y/w/h of every frame in the PNG.

The runtime path is:

```
SheetRecord (RON) ──► spec_from_record ──► CharacterSheetSpec ──► build_atlas ──► TextureAtlasLayout
```

`build_atlas` turns the spec into Bevy's `TextureAtlasLayout` — a flat
list of URects that the renderer indexes into via `flat_index(anim,
frame)`.

The user reported sprite **tearing** on the puppy slug, the kernel
guide, and the absurd general — animation frames showed the wrong
band of pixels, with strips of an adjacent row bleeding into the
visible silhouette.

The same generator pipeline was working fine for the goblin, pirate
raider, etc. The reporter's natural hypothesis ("regen vs reader
version mismatch") was wrong — both files were emitted in the same
session.

## Observable symptoms

- **Puppy slug:** Hit/Death frames render WALL_WALK / CEILING_WALK
  pixels because the post-filter `row_idx` shifted upward when those
  unrecognized rows were dropped from the spec.
- **Kernel guide & absurd general (toon-target generator):** Every
  frame is shifted ~4px because the generator emits 4px of padding
  between cells (row stride 93 when `frame_height = 89`; column stride
  124 when `frame_width = 120`), but the atlas builder assumes
  zero-padding grid math and samples 4px into the next cell.

## Root cause (two bugs in one read path)

`CharacterSheetSpec::build_atlas`:

```rust
for (row_idx, (_, row)) in self.rows.iter().enumerate() {
    for col in 0..row.frame_count {
        let x = self.label_width + col as u32 * self.frame_width;
        let y = self.y_offset + row_idx as u32 * self.frame_height;
        layout.add_texture(URect { min: (x, y), max: (x + w, y + h) });
    }
}
```

Two things go wrong here, both of which are invisible to the test
suite (atlas correctness is exercised only visually):

1. **`row_idx` came from `.enumerate()`, not from the authored
   row_index in the RON.** `CharacterAnim::from_name` silently
   drops any row whose name isn't in its match table (`wall_walk`,
   `ceiling_walk`, `chomp`, `dive`, `floor_slam`, `side_sweep`,
   `talk`, `interact`, etc.). Every surviving row after a dropped
   row got shifted UP by one `frame_height` in the atlas → sampled
   the wrong PNG row entirely.

2. **Grid math (`col * frame_width`) doesn't honor inter-frame
   padding.** The toon-target generator pads frames by 4 px to keep
   bilinear sampling safe, so cells live at x = 116, 209, 302, …
   (stride 93), not 116, 205, 294, … (stride 89). The runtime cell
   coordinates were off by `padding * col` pixels in x and
   `padding * row` in y.

Both bugs are forms of the same deeper anti-pattern: **the atlas
builder reinvents addressing math that the manifest already
encodes.** The RON's `rects: [{x, y, w, h}, ...]` IS the source of
truth — and was being ignored in favor of strided grid math that
happens to match SOME generated layouts and not others.

## Level B prompt

In Ambition's `crates/ambition_actors/src/presentation/character_sprites/sheets.rs`,
`CharacterSheetSpec::build_atlas` builds a `TextureAtlasLayout` from
a spec loaded from a `*_spritesheet.ron` manifest. The user reports
that the **kernel guide** and **absurd general** sprite sheets render
torn — each animation frame shows strips of the next row's pixels
along its edges. Other sprite sheets (goblin, pirate raider) render
correctly.

Here's the relevant slice of the kernel guide RON:

```ron
label_width: 112,
frame_width: 89,
frame_height: 89,
rows: [
    (animation: "idle",  row_index: 0, frame_count: 8,
     rects: [(x: 116, y: 4, w: 89, h: 89), (x: 209, y: 4, w: 89, h: 89), ...]),
    (animation: "walk",  row_index: 1, frame_count: 8,
     rects: [(x: 116, y: 97, w: 89, h: 89), ...]),
    (animation: "talk",  row_index: 2, frame_count: 6,
     rects: [(x: 116, y: 190, ...)]),
    (animation: "interact", row_index: 3, frame_count: 6,
     rects: [(x: 116, y: 283, ...)]),
]
```

And `build_atlas` currently does:

```rust
for (row_idx, (_, row)) in self.rows.iter().enumerate() {
    for col in 0..row.frame_count {
        let x = self.label_width + col as u32 * self.frame_width;
        let y = self.y_offset + row_idx as u32 * self.frame_height;
        layout.add_texture(URect { min: (x, y), max: (x + w, y + h) });
    }
}
```

The "goblin works, kernel guide tears" split is the diagnostic clue.
Identify **all** the addressing assumptions `build_atlas` is making
that hold for the goblin sheet but not the kernel guide / absurd
general / puppy slug, and propose a minimal fix.

## Expected answer

`build_atlas` makes three implicit assumptions, two of which are
violated by the user-reported sheets:

1. **Column stride = `frame_width`.** Violated by the toon-target
   generator, which pads frames by 4px → actual column stride is
   `frame_width + 4`. The fix: read each frame's authored `x` from
   `row.rects[col].x` instead of computing `label_width + col *
   frame_width`.

2. **Row index = position in `self.rows`.** Violated for any sheet
   with rows whose names aren't in `CharacterAnim::from_name`'s
   match table — those rows are filtered out of the spec, but the
   surviving rows still hold their AUTHORED `row_index`. The fix:
   propagate `row_index: u32` into `AnimRow` and use it for `y`
   computation (or read each frame's authored `y` from `rects`).

3. **Row stride = `frame_height`.** Violated by the same inter-frame
   padding — kernel guide's row pitch is 93. The fix: read each
   frame's `y` directly.

All three fixes reduce to one: **use the RON's per-frame `rects` as
the authoritative source for atlas-cell coordinates,** and treat the
grid-stride formula as a legacy fallback for sheets whose RON omits
the `rects` field. Update `AnimRow` to carry the rects (or a flat
`Vec<URect>` per row); update `build_atlas` to consume them; keep the
grid math behind an `if let Some(rects) = ...` branch for backward
compatibility.

The atlas image extent must also be computed from the union of all
rect maxes (not from `len() * frame_height`) so the atlas can cover
the padded region.

## Why this is a good candidate

Three reasons:

1. **The manifest had the right answer all along.** The RON `rects`
   field has carried the authoritative pixel coordinates for every
   frame since the schema was authored. The atlas builder ignored
   them and derived its own — which worked for sheets whose layout
   happened to match the implicit grid assumption. A future agent
   adding a new sprite generator with a different padding scheme
   will hit the same wall.

2. **`from_name` is a silent filter.** Filtering a Vec with
   `filter_map` and then iterating with `.enumerate()` is a common
   pattern, but the SECOND consumer treats the position as a stable
   index into the original Vec. The agent who wrote `build_atlas`
   assumed (silently) that `from_name` accepts every emitted row
   name. The toon-target additions (`talk`, `interact`,
   `celebrate`) and the puppy-slug additions (`wall_walk`,
   `ceiling_walk`) violate that assumption with no compiler error.

3. **The reporter's hypothesis was wrong, but his lieutenant
   ("benchmark candidate") instinct was right.** The user's
   first guess was "regen / version drift between PNG, YAML, RON"
   — natural because the symptom *looks* like a stale manifest.
   The actual fault is in the reader, not the data. This is exactly
   the failure mode worth journaling because the next person to
   debug a "sprite tearing" report will instinctively grep for
   regen mismatches and miss the reader-side bug.
