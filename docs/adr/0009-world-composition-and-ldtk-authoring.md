# ADR 0009: Use canonical world composition with LDtk as an authoring adapter

## Status

Accepted.

## Context

Ambition is intended to become a professional package for building large 2D games, not only a small sandbox. A room can be a designer-authored unit, a generated module, a streamed chunk, or a semantic area. The player experience should not be constrained by those authoring units.

The first correction driving this ADR is the central hub basement: the basement should be physically below the hub and reachable by dropping through a floor opening. It should not be a separate loading-zone room. More generally, if the player experiences movement as continuous traversal, the runtime should model it as one continuous active area even if the content was authored as multiple chunks.

External level-editing tools can make this easier, but they must not own Ambition's gameplay semantics. Ambition needs typed concepts, validation, and importer boundaries so generated content, hand-authored LDtk levels, and future tools can all target the same runtime model.

## Decision

Ambition distinguishes these concepts:

```text
RoomChunk / LevelChunk
  An authored unit: an LDtk level, generated module, RON fixture, or future editor section.

PlacedRoomChunk
  A chunk plus a world offset.

ActiveArea
  One continuous runtime play space composed from one or more placed chunks.

LoadingZone
  An intentional transition between active areas.

StitchedBoundary
  An authoring seam inside an active area; not a loading zone.
```

LDtk is the first external editor integration target. The sandbox should include a proof-of-concept LDtk project, an Ambition LDtk adapter, and an Ambition-specific validator. The adapter converts LDtk-authored chunks into Ambition's typed room manifest / engine world path.

The canonical gameplay model remains Ambition's typed data model, not raw LDtk JSON. `bevy_ecs_ldtk` is the Bevy ecosystem bridge for LDtk assets and tile/entity spawning workflows. The sandbox should spawn an `LdtkWorldBundle` and sync its `LevelSet` to the active Ambition area, while Ambition-owned conversion and validation decide what entities mean.

## Consequences

- LDtk files may be patched/generated by agents because they are JSON.
- LDtk entities must map to Ambition vocabulary such as `PlayerStart`, `Solid`, `OneWayPlatform`, `DamageVolume`, `EnemySpawn`, `BossSpawn`, `NpcSpawn`, and `DebugLabel`.
- The sandbox POC should compose `central_hub_main` and `central_hub_basement` into a single `central_hub_complex` active area.
- The player must be able to drop from the hub into the basement without transition state, fade, teleport, or spawn repair.
- Debug tooling should include an overview camera mode for large or stitched areas.
- Loading zones remain appropriate for intentional doors, portals, edges, or area transitions.
- Stitched chunk seams should eventually be validated for alignment, collision cracks, camera coverage, and spawn safety.

## Initial implementation target

The first implementation should be conservative:

1. Add `bevy_ecs_ldtk` as a dependency.
2. Add a sandbox `.ldtk` authoring file.
3. Add a Rust LDtk path that validates the Ambition subset and materializes chunks into runtime `RoomSpec` / `RoomSet` data.
4. Add a command-line validator script that can run without Cargo.
5. Update sandbox startup to prefer LDtk-authored rooms while retaining RON for abilities, tuning, and audio.
6. Spawn a `bevy_ecs_ldtk` `LdtkWorldBundle` and keep its `LevelSet` synchronized with Ambition room transitions.
7. Add F5 overview camera support for inspecting large/stitched active areas.

## Non-goals for the first implementation

- Making LDtk tile rendering the only runtime representation before Ambition collision, traversal, and gameplay validators are promoted to direct LDtk sources.
- External `.ldtkl` split-level files.
- World streaming or distance-based chunk activation.
- Replacing the existing custom platformer controller.
- Treating LDtk as the canonical gameplay schema.

## Review notes

Use `AMBITION_REVIEW(spatial)` near chunk composition, world/camera coordinate conversion, collision seams, and overview camera logic. These systems are likely to need property tests and visual diagnostics as the world grows.
