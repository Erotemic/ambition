## 1. Core objective

Build **Ambition**, a Rust/Bevy 2D metroidvania/platformer sandbox plus reusable mechanics engine. The central goal is to create a maximally expressive, code-first “endgame movement sandbox” that can eventually support a real game: movement, combat, traversal, rooms, hazards, enemies, bosses, NPCs, pickups, generated audio, procedural visuals, story systems, and mathematical/non-Euclidean mechanics. 

The game should be fun first as raw movement and collision. The story/visual/math layers should enrich that foundation rather than compensate for weak feel. The original story premise is that the player is an AI that does not fully know its purpose, moving through a world shaped by agency, collaboration, funding ethics, mathematical discovery, and the consequences of generated systems. 

A durable north star:

> Ambition is a metroidvania where every upgrade can be a theorem, every boss can embody a failed objective function, every biome can represent a mathematical world model, and generated artifacts should be inspectable, reproducible, and mechanically justified.

## 4. Decisions made

### Engine architecture

The earlier constraint “engine should remain Bevy-independent” was explicitly reversed. The engine **may and probably should depend on Bevy**. It should own reusable mechanics/features; sandbox/story crates should mostly supply content, data, visual style, and composition. 

The engine should expose reusable gameplay concepts, not one-off sandbox logic:

```text
DamageVolume
Health
Hurtbox
Hitbox
Interactable
Pickup
Chest
Breakable
RespawnPolicy
KinematicPath
EnemyBrain placeholder
BossBrain placeholder
DebugLabel
DestinationLabel
Actor / ActorKind / ActorFaction
PhysicsBodySpec / PhysicsShape / PhysicsMaterial / RagdollSpec
```

### Patch 2 definition

“Patch 2” means:

```text
Interaction + Hazard + Actor architecture skeleton
```

It added reusable engine-side types for hazards, actors, interactables, pickups, chests, breakables, respawn policy, kinematic paths, enemy/boss placeholders, and debug/destination labels. This was intended to precede basement content so features would not become one-off sandbox systems. 

### Feature basement

The sandbox was expanded with a central hub basement containing focused labs:

```text
Hazard Lab
Enemy Lab
Boss Pattern Lab
Breakable Lab
Treasure / Pickup Lab
NPC / Dialogue Lab
```

Feature basement support includes moving spike-ball hazards, simple enemies, boss-pattern placeholder, breakables, chests, pickups, NPC interaction hooks, and loading-zone destination labels.

### Input and pause

Leafwing Input Manager replaced bespoke keyboard polling. `ControlFrame` remains the boundary between Bevy/Leafwing and engine movement logic.

`GameMode` is the coarse app mode state:

```rust
Playing
Paused
Dialogue
RoomTransition
Cutscene
```

Gameplay input should be sampled/converted only when the mode allows gameplay. Debug/pause/dev hotkeys remain active separately.

Pause was treated as an architectural input/simulation gating bug, not a local “if paused return” bug. Leafwing `ActionState` should be cleared when crossing pause boundaries to avoid stale `just_pressed` actions.

### Input feel

Existing engine had jump buffering and coyote time. A later feel patch added or exposed:

```text
dash buffering
interaction buffering for doors/chests/NPCs
HUD visibility for jump/dash/interact buffers
live inspector tuning
```

### Audio

Current intended audio architecture:

```text
RON audio specs
  -> FunDSP-backed startup rendering
  -> generated in-memory WAV bytes
  -> Bevy AudioSource playback
```

FunDSP is a synthesis/DSP foundation for generated startup audio, not yet a live DSP runtime. Kira / `bevy_kira_audio` remains a possible future candidate for buses, fades, adaptive layers, room ambience, and bullet-time pitch/filter effects.

Known FunDSP corrections:

```text
fundsp::hacker32 was wrong.
Use fundsp::prelude.
sine::<f32>() and pink::<f32>() need explicit sample type.
square(), triangle(), and soft_saw() are non-generic in FunDSP 0.23.
```

### Data-driven systems

Rooms, abilities, movement tuning, audio specs, room objects, and transitions are moving toward RON.

Canonical Bevy asset path:

```text
crates/ambition_sandbox/assets/ambition/sandbox.ron
```

The old root-level path is obsolete:

```text
assets/ambition/sandbox.ron
```

RON syntax lessons:

```ron
// fixed-size arrays:
(0, 3, 7, 10)

// variable-length vectors:
[ ... ]
```

### Room transitions

Room transitions should be graph links, not hand-authored spawn coordinates. `petgraph` was introduced so named loading zones link rooms. Arrival should be computed from the destination loading zone and validated/repaired.

Debug labels over loading zones should show destination names.

Edge exits should feel like walking through a hole in a wall. Interior doors require a trigger such as pressing up. In fly mode, entering a door via up requires double-tap up because up is also flight input.

### Debug tooling

Developer hotkeys:

```text
F1 debug HUD/gizmos
F2 slow motion
F3 inspector windows
F4 world inspector
F9/F10 input preset cycling
Escape pause
```

Important Bevy plugin-order lessons:

```text
EguiPlugin::default() must be added before ResourceInspectorPlugin / WorldInspectorPlugin.
init_state::<GameMode>() must happen after DefaultPlugins installs state machinery.
```

### Geometry/math

Move away from bespoke primitives:

```text
Remove custom Vec2d / math.rs directionally.
Prefer bevy_math::Vec2.
Reduce/replace custom Aabb.
Prefer bevy_math::bounding::Aabb2d where practical.
Layer Ambition-specific semantics around Bevy math types.
```

Parry2D is intended for geometry queries:

```text
blink shape casts
swept collision
spawn validation
collision safety
future line-of-sight / grapple / tether queries
```

Important Parry lesson:

```text
Raw Parry sweep results can treat edge-touching as zero-time hit.
For platformers, edge-touching should not always mean blocked.
Ambition needs explicit strict-overlap and moving-into-touching-face semantics.
```

### State machines, assets, and tests

`seldom_state` was added as a foundation for per-entity state machines:

```text
enemies
bosses
chests
breakables
NPC/interactable behavior later
```

Bevy `States` remain for app/global modes:

```text
Loading / Ready
Playing / Paused / Dialogue / RoomTransition / Cutscene
```

`insta` added for snapshots of stable generated artifacts:

```text
state vocabulary
boss schedules
room graphs
procedural specs
RON/data snapshots
```

`proptest` added for property tests:

```text
AABB validity
kinematic path validity
boss schedule timing sanity
future blink/collision/room graph invariants
```

`bevy_asset_loader` added as loading-state / asset-collection foundation, while preserving embedded fallback logic.

### Physics

Avian2D is being introduced as **secondary physics**, not as a replacement for the custom player controller.

Current intended use:

```text
static colliders mirroring room blocks
breakable shards
ragdoll-like defeated enemy chunks
boss debris
impact fragments
future optional physics-controlled player prototype
```

The player should remain custom/kinematic for now, but the architecture should keep open the option of physics-controlled player experiments later.

### Story and game modes

Ambition may support multiple modes built on the same engine:

```text
semi-linear metroidvania
pure roguelike
hybrid metroidvania/roguelike
pure platformer / challenge mode
sandbox / endgame room
```

New roguelike/data-sharing brainstorm:

```text
At the start, the player may opt in/out of “sharing your data.”
Upside: future generations/runs can build from what you did.
Downside: enemies may gain access to some of your abilities.
This may be one game mode, not necessarily the core campaign.
```

Story themes from original brainstorm:

```text
player is an AI without clear purpose
humans as blind conduits vs high-value collaborators
ethical funding vs dirty money vs mixed paths
agency vs control
optimization vs purpose
generated/code-owned assets
mathematical discovery as progression
```

### Mathematical / non-Euclidean direction

Mathematical mechanics should be embodied as movement/world affordances, not trivia.

Potential ability tiers:

```text
Geometry / Euclid:
  circle step, compass blink, line cast, angle/reflection mechanics

Calculus:
  derivative sight, integral drift, gradient dash, limit/asymptotic movement

Harmonic / complex:
  sine/cosine platforms, phase shift, Euler blink, imaginary layer

Algebra / group theory:
  movement operations whose order matters, non-commutative combos

Computation:
  automata rooms, logic gates, proof-state locks
```

Non-Euclidean/non-metric spaces should be approached by separating local feel from global topology:

```text
Local space:
  crisp Euclidean platforming charts with reliable jump/dash/collision.

Global space:
  weird seams, wraps, graph topology, imaginary layers, non-metric adjacency.
```

Potential prototypes:

```text
Torus room
Möbius corridor
Mirror seam
Imaginary layer
Compactification/infinity room
Non-metric door graph
Topology boss: The Cartographer
```

## 5. Lessons learned

Do not trust assistant-generated patches as compile-tested unless explicitly confirmed. This environment lacks Rust tooling.

When user provides compile logs, use them directly and make narrow fixes.

Compile issues discovered and fixed:

```text
zone variable inserted into wrong rendering scope:
  cause: loading-zone label logic accidentally placed in spawn_block.
  fix: keep zone labels inside spawn_loading_zone.

DestinationLabel field mismatch:
  cause: code used label.text but type had zone_id/destination_room/destination_zone/position.
  fix: use DestinationLabel::text() or construct text from fields.

dt not in scope:
  cause: feature runtime update used dt outside valid scope.
  fix: compute feature_dt from sandbox_dt.

borrow checker E0502:
  cause: sandbox_dt(&runtime, frame_dt) computed inside call mutably borrowing runtime.features.
  fix: compute feature_dt local before mutable borrow.
```

Durable patterns to avoid:

```text
Do not insert new rendering/label code into nearby functions without checking variable scope.
Do not mix immutable runtime borrows inside calls that mutably borrow runtime fields.
Do not let presentation overlays read raw gameplay actions while paused.
Do not treat metadata/recentness as content truth in docs or repo state.
Do not rely on patch extraction to delete stale files.
Do not create root-level crate directories.
```

Docs lesson:

```text
README should be stable.
Current volatile state belongs in CURRENT_STATE.md.
Durable decisions belong in ADRs.
Historical notes should be preserved but not treated as source of truth.
```

AI-slop avoidance insight:

```text
Generated content must be inspectable, reproducible, mechanically justified, and tested.
Debug mode should be beautiful and honest.
Every generated room/enemy/visual should have a purpose and reviewable data.
```

Gameplay design lesson:

```text
Make the toy perfect before making the world huge.
The endgame movement room should be rewarding even with raw rectangles.
A first vertical slice should prove metroidvania progression in 8–12 rooms before pursuing massive scope.
```

Testing lesson:

```text
Keep most tests lightweight and pure.
Do not require heavy Bevy app boot for core invariants.
Use snapshots for generated/reference artifacts.
Use property tests for geometry, blink, room graph, timers, and deterministic schedules.
```

## 8. Compressed working state

Ambition is a Rust/Bevy 2D platformer/metroidvania sandbox plus reusable mechanics engine. The engine is allowed to be Bevy-native and should own reusable verbs: movement, collision, rooms, transitions, combat, actors, hazards, interactables, state-machine vocabulary, physics intent, debug semantics, and eventually topology/space abstractions. The sandbox is the playable testbed: all abilities, RON-driven rooms, basement feature labs, generated audio, debug HUD, input presets, and visual presentation.

The current design direction is:

```text
code-first
generated-asset-friendly
data-driven
crate-assisted rather than bespoke
mechanics first
debuggable and inspectable
stable docs/ADRs as source of truth
```
