* Build **Ambition**: a Rust/Bevy 2D metroidvania/platformer sandbox plus reusable mechanics engine, centered on expressive movement, reusable gameplay systems, generated/inspectable assets, and eventually mathematical/non-Euclidean mechanics.

## 4. **Decisions made**

* Engine may/probably should depend on Bevy.

  * Earlier “engine must be Bevy-independent” was explicitly reversed.
  * Rationale: engine should own reusable Bevy-native mechanics/features.

* Engine should expose reusable gameplay concepts, not one-off sandbox logic:

  * `DamageVolume`
  * `Health`
  * `Hurtbox`
  * `Hitbox`
  * `Interactable`
  * `Pickup`
  * `Chest`
  * `Breakable`
  * `RespawnPolicy`
  * `KinematicPath`
  * `EnemyBrain`
  * `BossBrain`
  * `DebugLabel`
  * `DestinationLabel`
  * `Actor`, `ActorKind`, `ActorFaction`
  * `PhysicsBodySpec`, `PhysicsShape`, `PhysicsMaterial`, `RagdollSpec`

* “Patch 2” means:

  * Interaction + Hazard + Actor architecture skeleton.
  * Intended to precede basement content to avoid one-off feature systems.

* Leafwing Input Manager replaced bespoke keyboard polling.

  * `ControlFrame` remains boundary between Bevy/Leafwing input and engine movement logic.

* Pause/input architecture:

  * Gameplay input sampled/converted only when mode allows gameplay.
  * Debug/pause/dev hotkeys separate.
  * Leafwing `ActionState` should be cleared across pause/dialogue boundaries to avoid stale `just_pressed`.

* Interactions:

  * Raw single-press Up should not activate doors/NPCs/chests.
  * Use explicit `Interact`, or deliberate double-tap Up fallback.
  * Rationale: Up is too useful as a movement direction; accidental door entry is easy.

* Audio:

  * Current intended architecture:

    ```text
    RON audio specs
      -> FunDSP-backed startup rendering
      -> generated in-memory WAV bytes
      -> Bevy AudioSource playback
    ```

  * FunDSP is startup synthesis foundation, not live DSP runtime.

  * Kira / `bevy_kira_audio` remains future candidate for buses/fades/adaptive layers/ambience/bullet-time pitch/filter effects.

* Geometry/math:

  * Move away from custom `Vec2d` / bespoke math.
  * Prefer `bevy_math::Vec2`.
  * Reduce/replace custom AABB.
  * Prefer `bevy_math::bounding::Aabb2d` where practical.
  * Layer Ambition semantics around Bevy math types.

* Parry2D:

  * Intended for blink shape casts, swept collision, spawn validation, collision safety, future line-of-sight/grapple/tether.
  * Platformer semantics need strict-overlap/moving-into-touching-face handling because raw Parry sweep can treat edge-touching as zero-time hit.

* Room transitions:

  * Use graph links, not hand-authored spawn coordinates.
  * `petgraph` introduced for named loading-zone links.
  * Arrival computed from destination loading zone and validated/repaired.
  * Debug labels over loading zones show destination names.
  * Edge exits should feel like walking through holes.
  * Interior doors require `Interact` or deliberate double-tap Up.

* State machines/testing:

  * `seldom_state` foundation for per-entity state machines:

    * enemies, bosses, chests, breakables, NPC/interactables.
  * Bevy `States` remain for app/global modes.
  * `insta` for stable generated artifacts:

    * state vocabulary, boss schedules, room graphs, procedural specs, RON/data snapshots.
  * `proptest` for:

    * AABB validity, kinematic path validity, boss timing sanity, future blink/collision/room graph invariants.
  * `bevy_asset_loader` added as loading-state/asset-collection foundation while preserving embedded fallback logic.

* Physics:

  * Avian2D introduced as **secondary physics**, not a replacement for custom player controller.
  * Use for:

    * static colliders mirroring room blocks.
    * breakable shards.
    * defeated enemy chunks.
    * boss debris.
    * impact fragments.
    * optional future physics-controlled player prototype.
  * Player remains custom/kinematic for now.

* Sandbags:

  * Should not spawn globally in every room.
  * Should be placeable room-authored objects.
  * Should reuse same code path as enemies as much as possible.
  * Current direction: sandbags as `EnemySpawn` with custom non-hostile brain:

    * `Custom("sandbag_infinite")`
    * `Custom("sandbag_finite")`

* Dialogue/NPC/merchant architecture:

  * Use `bevy_yarnspinner` as long-term dialogue content/runtime direction.
  * Use `bevy_material_ui` as UI component direction.
  * Keep first UI custom/stable while wiring toward Yarn events.
  * NPC interactions emit dialogue requests instead of only banners.
  * Merchant is design stub pending real inventory/wallet/effects.
  * ADR added/proposed: `docs/adr/0008-dialogue-and-commerce-architecture.md`.

* Basement/world:

  * User corrected previous interpretation:

    * Basement should be physically below central hub, not merely a separate room.
  * Do not fake this with another loading-zone room as final direction.
  * Need future architecture patch: floor openings / stitched room chunks / bigger hub.

## 5. **Lessons learned**

* Do not trust assistant-generated patches as compile-tested unless explicitly confirmed.

  * Environment lacks Rust tooling.
  * User compile logs are authoritative.

* Zip handoff lesson:

  * Several patch links/payloads were wrong or stale.
  * Need verify:

    * file exists in `/mnt/data`.
    * `unzip -t` passes.
    * archive has exactly one top-level directory for `--strip-components 1`.
    * archive contains the actually edited files/lines.
  * Bad examples:

    * health overlay zip had wrong archive shape; user had to manually fix.
    * dialog compile fix v1 still contained old `BorderColor(...)`.
    * fresh v2 had verified `BorderColor::all(...)` and `_flying`.

* Compile issue: missing trait import.

  * Error: no `half_size()` / `center()` for `Aabb2d` in `physics.rs`.
  * Cause: extension trait not in scope.
  * Fix: `use ambition_engine::AabbExt;`
  * Later stale import in `main.rs` caused warning; remove where unused. 

* Compile issue: Bevy system tuple `.chain()` failure.

  * Error: tuple with systems could not call `.chain()`.
  * Durable cause: `sandbox_update` signature grew too large / invalid as Bevy system after adding another resource parameter.
  * Fix: remove `PhysicsSandboxSettings` direct parameter from `sandbox_update`; store settings copy in `SandboxRuntime` at setup and use there. 

* Compile issue: Bevy 0.18 `BorderColor`.

  * Error: `BorderColor(...)` invalid; it is a struct in Bevy 0.18.
  * Fix: `BorderColor::all(Color::srgba(...))`.
  * Also warning: `LoadingZone::hint(flying)` unused; fix `_flying`.

* Known FunDSP corrections:

  * `fundsp::hacker32` was wrong.
  * Use `fundsp::prelude`.
  * `sine::<f32>()` and `pink::<f32>()` need explicit sample type.
  * `square()`, `triangle()`, `soft_saw()` are non-generic in FunDSP `0.23`.

* Bevy plugin-order lessons:

  * `EguiPlugin::default()` before `ResourceInspectorPlugin` / `WorldInspectorPlugin`.
  * `init_state::<GameMode>()` after `DefaultPlugins` installs state machinery.

* Debug overlay lesson:

  * Initial feature combat volumes were not drawn.
  * Added drawing for hazard damage, enemy body/hurt volumes, attack telegraphs, active hitboxes, boss body/attack volumes.
  * One-pixel gizmo health bars were too easy to miss and could not show text.
  * Better: world-space overlay entities with background bar, fill bar, and `current/max` text.

* Combat feel lesson:

  * Enemies need attack telegraphs; invisible/instant hitboxes feel unfair.
  * Body-contact damage should be explicit and visible.
  * Sandbags should be non-hostile even if slashable.
  * Player needs damage modes:

    * `SafeRespawn` for lava/fall hazards.
    * `Knockback` for enemy/boss hits.
    * Death: room-start respawn with full HP for now.

* Boss movement lesson:

  * Prior movement used current boss position as feedback near player and produced sign-flip/flicker/two-views.
  * Boss movement must:

    * use stable spawn/anchor reference.
    * be speed-limited.
    * respect collision/platforms/room bounds.
    * avoid per-frame respawn/sync instability.
  * Boss/player are custom/kinematic; the observed boss issue was likely not Avian directly.

* Avian warnings:

  * Runtime warnings seen:

    ```text
    Tried to wake body ... that does not exist
    ```

  * Attempted/proposed mitigation:

    * retire physics bodies by adding `RigidBodyDisabled` + `ColliderDisabled` before despawn.
    * room transitions retire Avian entities before despawning.
    * debris expiration disables briefly before despawn.

  * Still unresolved; need confirmation from latest run logs. 

* Feature/runtime anti-patterns:

  * Do not insert new rendering/label logic into nearby functions without checking variable scope.
  * Do not mix immutable `runtime` borrows inside calls mutably borrowing `runtime.features`.
  * Do not let presentation overlays read raw gameplay actions while paused/dialogue.
  * Do not rely on patch extraction to delete stale files.
  * Do not create root-level crate dirs.
  * Do not treat metadata/recentness as content truth.

* Previous compile fixes:

  * Loading-zone `zone` variable accidentally inserted into wrong rendering scope; keep zone labels in `spawn_loading_zone`.
  * `DestinationLabel` field mismatch; use `DestinationLabel::text()` or construct from actual fields.
  * `dt` not in scope in feature runtime; compute `feature_dt`.
  * Borrow checker E0502 from computing `sandbox_dt(&runtime, frame_dt)` inside mutable borrow; compute local before mutable borrow.

* Design lessons:

  * Make the toy perfect before making the world huge.
  * Endgame movement room should be rewarding with raw rectangles.
  * First vertical slice should prove progression in 8–12 rooms before huge scope.
  * Generated content must be inspectable, reproducible, mechanically justified, and tested.
  * Debug mode should be beautiful and honest.
  * Every generated room/enemy/visual should have a purpose and reviewable data.
