## 1. The headline: the engine/content oracle regressed silently

decomposition.md exit criterion 2 says "the named-content grep over engine
crates hits zero." At HEAD that is **false in spirit**: the enforcement is a
set of narrow per-crate identifier denylists, and named Ambition content has
re-accumulated in `crates/` through gaps between the lists. This is the exact
failure mode the workspace-policy package exists to prevent, and it is the
highest-value cleanup available ‚Äî not because any single leak is large, but
because each one is a place where a second game (or a weak agent) is forced to
edit core.

## 2. The second monolith: `ambition_app` is ~60% engine machinery

The decomposition campaign killed the actors monolith and the residue queue is
honest ‚Äî but the same mass quietly re-formed a tier up. `ambition_app` (22.3k)
is supposed to be thin wiring; measured, it is:

- **`menu/` ‚Äî 10.1k (‚âà5.7k non-test).** The backend-agnostic menu HOST stack:
  `MenuPageModel` builder from live game state, dispatcher, item effects, and
  BOTH presentation hosts (Lunex kaleidoscope + Bevy-UI grid). Grep confirms
  ZERO named Ambition content in it. The E1e ruling kept the host stack
  app-side because it "couples up to items/player/sfx" ‚Äî but those are all
  *engine* crates; the coupling never blocked a downward home, and any second
  game must otherwise rewrite ~5.7k LOC of pause/inventory hosting.
  ‚Üí **Proposed: `crates/ambition_menu_host`** (above `ambition_menu` +
  `items` + `sim_view` + `persistence`; below the apps). This revisits E1e
  with new evidence; flagged as a Jon confirmation, not a unilateral reversal.
- **`rl_sim/` ‚Äî 1.7k.** The programmatic-sim/gym seam (`SandboxSim` step API,
  `AgentAction`/`AgentObservation`/reward), explicitly designed for RL/fuzz/
  replay/PyO3 ‚Äî trapped where no other game (and no external tool) can reach
  it without depending on Ambition's app. This is load-bearing for the RL
  north star. ‚Üí **Proposed: `crates/ambition_sim_harness`** with the app
  passing its plugin composition in through a builder seam.
- **`dev/` ‚Äî 2.8k.** Debug overlay + gizmos + FPS overlay + portal inspector.
  architecture.md already gives `ambition_dev_tools` the "[the workbench] may
  see everything; nothing sees it" charter. ‚Üí move overlays into
  `ambition_dev_tools` behind an `overlay` feature.
- **`app/world_flow/room_flow.rs` (494)** ‚Äî the room-transition APPLY
  composer, an E5 "slot" that every game re-implements. ‚Üí runtime-adjacent
  home once a second consumer exists (Mary-O completion will force this).

Two more app-layer findings of the same species:

- **The facade carries logic.** `crates/ambition/provider.rs` (828) is the
  provider preparation/session-build ENGINE (catalog fragments, the 10-step

## 4. Unifications (the emergence agenda)

- **U1 ‚Äî Boss onto the moveset (the big one).** The moveset is confirmed as
  THE vocabulary for player/enemy melee+ranged+special; the boss stack
  (`characters::brain::boss_pattern` ~5.2k + `actors::boss_encounter`
  attack_geometry ~2k) is the last parallel combat vocabulary, and the moveset
  code already names the fold as its intended future. This SUBSUMES most of
  the boss/actor tangle: `boss_encounter`‚Üî`features` is bidirectionally welded
  through `features/ecs/boss_clusters.rs`, and the encounter phase-machines
  (`BossEncounterPhase` vs `EncounterPhase`) are the two-schema problem
  tracks.md already queues as #1. **Order matters: unify BEFORE any boss
  carve** ‚Äî carving first would freeze the second vocabulary into a crate
  boundary.
- **U2 ‚Äî One timeline, three scripts.** `CutsceneScript` (399-LOC crate),
  `EncounterScript` (the ruled ONE timeline authority), and Yarn are three
  authored beat/trigger/effect vocabularies. Cutscene beats (Wait/Dialogue/
  CameraPan/Fade/SetFlag/Banner) are expressible as encounter-timeline effects.
  Proposal for a Jon ruling: fold cutscene onto the encounter timeline (Yarn
  stays ‚Äî words are genuinely a different domain), or explicitly bless three
  vocabularies in architecture.md so agents stop half-merging them.
- **U3 ‚Äî UI micro-consolidation.** `ambition_inventory_ui` (149 LOC) is one
  model + one enum; fold into `ambition_ui_nav` or `ambition_menu`. Decide
  whether settings renders THROUGH `MenuPageModel` (one menu framework) or
  stays a parallel IR. Reclassify `ambition_game_shell` in the docs as shell/
  routing tier, not menu cluster.
- **U4 ‚Äî intra-actors tangle repair (ruling-compatible).** Move
  `features/ecs/boss_clusters.rs` (429) beside its consumers in
  `boss_encounter/` to collapse the crate's worst internal bidirectional edge;
  collapse the thin facades (`projectile/`, `enemy_projectile/`, `items/`
  glue, `platformer_runtime`, `physics`, `effects`) into their owning modules
  per anti-god rule 3. No new crate; pure navigability.

## 5. What we deliberately do NOT propose

- **No `ambition_actors` crate split.** The 2026-07-10 ruling stands: the
  rebuild tower above actors dominates compile time, and splitting the actor
  sim re-forks the unification. The leaf modules the anatomy found (`dev/`,
  `menu/`, `gravity/`, `dialog/` ‚Äî ~3.5k combined, zero reverse-deps) are the
  already-enumerated residue queue, not a new carve; they dissolve into their
  owning crates when their preconditions land. IF Jon wants to revisit, the
  only candidate with a real story is a boss crate ‚Äî and only AFTER U1.
- **No `engine_core` split.** It measured as a coherent kernel with clean
  internal clusters (geometry/cast, reference frames, movement, body clusters,
  world IR, control-frame vocabulary) and touching it rebuilds in ~5s. The
  extractable units (collision library, reference-frame crate) are real but
  buy nothing today. Two cosmetic notes only: `config.rs` window constants are
  presentation values in a sim crate, and the `DEFAULT_*` tuning constants are
  Ambition's numbers as universal defaults (acceptable; content overrides).
- **No sim_view inversion.** Publishing view types below actors and having
  actors push into them splits types from their builders; the current pull
  model quarantines the actors dep in exactly one crate. Recorded as a
  considered-and-rejected tradeoff.
- **No audio-triple merge.** `sfx_bank`/`sfx`/`audio` layering is justified
  (pure format / contract / Kira+director). `sfx_bank`‚Üí`sfx` is the only
  tenable merge and it isn't worth the churn.
- **No mass UI merge.** The nav-primitives / framework / specific-menus
  layering is already right; only U3's micro-folds apply.
