# Audio and particle notes

## Generated audio

The sandbox now creates short sound effects at startup from symbolic synth
recipes. Each recipe defines waveform, frequency sweep, duration, envelope,
volume, and optional noise. The code renders stereo 16-bit PCM into an in-memory
WAV buffer and gives that buffer to Macroquad's audio loader.

This follows the same architectural shape as the older pygame prototype:

1. identify the event by a small sound id;
2. render or load a cached/generated PCM/WAV representation;
3. play the resulting runtime sound object;
4. keep sound design as data, not imported assets.

For the current sandbox, the useful cues are jump, double jump, dash, slash,
hit, pogo, reset, dummy death, and dummy respawn.

## Particle system

The current particle system is intentionally a tiny CPU-side list of structs:
position, velocity, age, lifetime, radius, color, gravity, drag, and kind. The
renderer draws circles and line shards using Macroquad primitives.

This is enough for movement feedback and early feel testing. It is not meant to
be the final visual effects architecture.

## Framework decision

Macroquad is still adequate for this phase. It already gives Ambition:

- input polling;
- primitive rendering;
- audio playback from generated bytes;
- a small dependency surface;
- fast iteration for code-first prototypes.

A framework upgrade is not necessary for small generated SFX or hundreds of
simple particles. Reconsider the renderer/engine choice if Ambition needs GPU
particle fields, shader graphs, editor tooling, complex animation graphs,
large-scale asset management, or a more formal ECS-driven architecture.

## Macroquad audio feature

The sandbox depends on `macroquad = { version = "0.4.14", features = ["audio"] }`. Without that feature, Macroquad compiles a no-op audio module and prints `macroquad's "audio" feature disabled` whenever a sound is requested.
