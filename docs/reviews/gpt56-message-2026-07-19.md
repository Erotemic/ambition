## 3. The rollback forcing function is much narrower than advertised

This is the most important newly discovered issue.

`game/ambition_app/tests/rollback_coverage.rs` says it audits every component on simulated entities and prevents new rollback state from slipping through. In practice, it queries only entities carrying `FeatureSimEntity`.

That population is not equivalent to the rollback population:

* The primary controlled body deliberately does not carry `FeatureSimEntity`.
* The rollback registry anchors all `BodyKinematics` entities.
* It also covers room/session roots, projectiles, encounter lifecycle entities, ground items, placed portals, gravity switches, and other families that are not necessarily represented by the queried marker.
* The test therefore cannot detect a new component added only to the primary body or to a root-only rollback entity.

Its fixture also boots one default simulation and advances eight idle frames. That does not guarantee the existence of bosses, held projectiles, portal transit state, demo-specific state, active encounters, or many optional ability families.

The waiver mechanism is additionally too broad. Waiving entire prefixes such as `ambition_cutscene::`, `ambition_dialog::`, `ambition_load::`, or `ambition_input::` means future authoritative state placed in those crates could be silently classified as harmless.

Finally, the comment that resources must be reviewed by hand is not technically necessary. The repository already uses `World::iter_resources()` in `game/ambition_app/tests/plugin_minimal_app.rs`. Resource coverage can be computed, even if the classification rules need a separate registry.

I think we should redesign this guard before relying on it as closure for Track 0:

* Audit every entity carrying the GGRS rollback marker, or construct the union of every registered anchor family.
* Assert that every expected family contributes at least one entity, so an accidentally absent fixture cannot produce a vacuous green result.
* Add scenario fixtures that instantiate dynamic and optional state rather than relying on eight idle frames.
* Audit resources separately against registered, derived, immutable, host-only, and presentation classifications.
* Make waivers type-specific wherever possible rather than waiving a whole crate namespace.
* Add an explicit assertion that every rollback-relevant body has a `SimId` before systems such as target selection consume identity.

The last item also bears on `combat/targeting.rs`. Candidates without `SimId` still fall back to raw `Entity` ordering. That may be unreachable in production because `ensure_sim_id` runs before `WorldPrep`, but the correct contract is probably “a rollback body without semantic identity is forbidden,” not “it is snapshot-irrelevant.” Please verify that contract and add a forcing test if it is intended.

## 4. Our test suite does not currently prove the feature compositions we think it proves

`scripts/run_tests.py` skips feature jobs for:

* `ambition_app`
* `ambition_actors`
* `ambition_host`
* `ambition_menu`
* `ambition_runtime`

The justification says those features gate no tests. The portal work has already disproved that assumption for `ambition_host`: with `portal_render`, five demo-shell tests panic because `PlatformerHostPlugins` installs portal-observation systems without the plugin that registers their messages.

That reveals two broader problems:

First, `cargo test --workspace` feature-unifies the graph. It can hide undeclared or accidentally transitive feature dependencies—the exact category of defect uncovered in the menu/build cleanup.

Second, features are functioning as product-persona composition controls, but we do not have a small explicit matrix of supported personas.

I do not recommend testing every feature combination. I recommend defining a curated set of supported compositions, probably including:

* Minimal headless simulation.
* Default workspace.
* Host with input.
* Host with portal mechanics and portal presentation.
* RL/simulation API.
* Visible desktop app.
* Web compile persona.
* Android compile persona.
* Each demo app’s intended composition.

Every feature that remains should name at least one supported persona and at least one check that exercises it. Features with no distinct persona or test value should be deleted or made honestly always-on.

The portal failure also suggests a plugin-composition rule: a plugin should either install its prerequisites, encode them in a plugin group, or safely disable itself when its required resources/messages are absent. Hidden “this feature only works if some content plugin happened to run first” contracts will continue creating combinatorial fragility.

## 6. Repository tooling itself needs executable health checks

`scripts/run_source_analysis.sh` is currently broken:

* It runs `cargo check --message-format=json` and then tries to execute `.agent/reports/cargo-check-warnings.md` instead of redirecting or processing the output.
* It calls `tools/ecs_inventory.py`, while the live script is `scripts/ecs_inventory.py`.

This is exactly the kind of navigability defect a holistic review should catch. Documentation and generated reports are only trustworthy if their advertised regeneration commands are executable.

I suggest a lightweight repository-tool smoke test that checks the help or dry-run path for maintained scripts and validates referenced script paths. We should not build a large scanner framework, but executable documentation needs at least one forcing function.

There is also an archive hygiene defect: the source archive opens with `.agent/manifest.yaml` already modified. `generate_agent_index.py` deliberately removes volatile provenance from the tracked manifest and writes it to `generation_stamp.json`, but `archive_agent_source.py` adds `generated_from_commit` and `generated_at` back into the tracked manifest in the staged archive. Because the archive includes `.git`, every extracted agent checkout appears dirty immediately.

Archive provenance should remain in `.agent/source_archive_manifest.yaml` or an ignored generation stamp. A fresh agent archive should have a clean `git status`.

## 7. Our planning surfaces are beginning to reproduce the bloat they are supposed to control

The current documents contradict their own ownership rules:

* `docs/planning/tracks.md` calls itself a live queue rather than a completion ledger, but it retains substantial landed narratives for Tracks 0, 2, 8, and 9.
* `dev/journals/code_smells.md` keeps many resolved items inside the `Open` section.
* `AGENTS.md` says it should remain short, stable, and routing-oriented, but its ONE BODY material is now packed into a 2,838-character line containing detailed live implementation paths.
* The 180-line AGENTS limit was satisfied partly by compressing prose onto huge lines. The checker itself acknowledges that line count is gameable, but the hard AGENTS rule still rewards that behavior.

I propose a stricter ownership model:

* `status.md`: present architectural truth only.
* `tracks.md`: open task cards only—ID, state, dependency, owner grade, exit oracle, and a few evidence links.
* Archived reviews: full evidence and historical narratives.
* `code_smells.md`: a compact active index; resolved entries moved to a resolved/history section or archive.
* `AGENTS.md`: evergreen operating principles and navigation links, not a live implementation map.
* Focused concept documents: detailed invariants such as the unified controlled-actor model.
* Size warnings based on words/characters and pathological paragraph length, not only line count.

This is not cosmetic. Agents currently have to reconcile the same work across status, tracks, roadmap, smell log, review archive, and subsystem plans. Every duplicated completion statement is another place for reality to drift.

## 8. “Rebase before committing” is not enough for parallel-agent integration

The stale-base regression was not an isolated Git mistake. It was a predictable result of several long-running agents modifying overlapping areas.

We need an explicit intake protocol:

1. Every agent task records its base SHA.
2. Before applying or committing its result, compare the patch’s touched paths against `BASE..HEAD`.
3. Any overlap requires replaying or rebasing the work onto current HEAD and reviewing the resulting semantic diff.
4. Relevant tests run after that replay, not merely on the agent’s original base.
5. Long-running tasks use branches or worktrees rather than editing shared main directly.
6. The handoff records the base SHA, final verified HEAD, touched ownership areas, and validation performed.

A small script can automate the path-overlap check without pretending to solve semantic merges. This directly addresses the failure mode your review found.

## 9. We need explicit complexity budgets, but not arbitrary crate-count targets

The repository now has 51 workspace crates, a deep serialized dependency spine, many plugin groups, and a substantial feature matrix. I agree with the review that raw file or crate size is not sufficient reason to carve more crates.

Instead, every new crate, feature, plugin, or cross-cutting framework should answer:

* What stable responsibility does it own?
* What dependency direction does it improve?
* Which supported persona benefits?
* What existing code or dependency does it allow us to remove?
* What is its rollback/derived/presentation classification?
* What plugin or schedule prerequisites does it introduce?
* What test proves the composition?
* Is this serving at least two real consumers, or are we generalizing a single use prematurely?

For existing work, prefer deletion, convergence, and role eviction over adding adapters and registries. A refactor that only relocates complexity without shrinking dependency or conceptual surface should not be counted as an architectural win.

The 324 GB target directory and 78 GB incremental cache also deserve an operational guardrail. We need not disable incremental compilation blindly, but setup or reporting tooling should warn when build artifacts cross a documented threshold and provide a canonical cleanup command.
