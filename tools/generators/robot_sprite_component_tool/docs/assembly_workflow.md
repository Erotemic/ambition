# Assembly workflow

The assembly layer consumes the refined component slices produced by
`tools/robot_asset_tool.py` and builds labeled, row-packed spritesheets.

## Commands

```bash
python tools/robot_asset_tool.py build metadata/robot_components.rough.yaml
python tools/robot_rig_sheet.py spritesheet examples/robot_rig_job.yaml output/assembled/robot_assembled_spritesheet.png
```

The main output intentionally keeps the row labels.  The labels are rendered by
Pillow, not generated by an image model, so they stay crisp.

## What the assembler fixes

The initial procedural assembly exposed several rig problems.  The current
assembler addresses the first batch of them:

- hands use a smaller per-part scale, so the idle/walk/run poses no longer read
  as giant boxing gloves;
- arm, hand, leg, torso, head, and face each have independent scale
  multipliers instead of one global scale;
- frames are composed against a stable root/ground anchor instead of visual
  centering, reducing in-game jitter;
- run and dash now have distinct silhouettes and effect sizes;
- hit remains a recoil/stagger only and does not reuse death poses;
- death has a separate collapse path;
- teleport has charge, vanish-hold, and reappear phases;
- slash effects were reduced and shifted so they originate closer to the hand
  and do not collide with neighboring cells;
- each frame is audited for alpha bounds and near-edge clipping.

## Frame QA

The generated YAML manifest contains per-frame `bbox`, `margins`, `root`, and
`warnings`.  The top-level `qa_warnings` list is empty when every nontransparent
pixel has at least the required edge margin.

The default job currently renders at 192x192 per frame with a 58px label column
and 2px gutters.  This is intentionally a little larger than the minimum because
jump, dash, slash, and death all need extra clearance.

## Tuning notes

Most visual tuning lives in `animation_pose()` in `tools/robot_rig_sheet.py`.
The code is deliberately structured so this can later move to YAML-driven pose
recipes.  The important variables are:

- `root_offset`: stable per-frame root motion;
- `torso_angle`, `head_angle`, limb angles: pose timing;
- per-part scales: `torso_scale`, `head_scale`, `arm_scale`, `hand_scale`,
  `leg_scale`, `face_scale`;
- `fx_behind` / `fx_front`: effect sprite, offset, scale, angle, and opacity.

Keep labels in the preview / game sheet unless a downstream engine export
requires a no-label atlas.  The metadata already records cell locations, so a
separate no-label export can be added later without changing pose logic.


## Visor overlay alignment

Do not paste expression sprites as independent face plates.  The assembler detects
the dark visor bbox on the selected head sprite, repaints that visor in place,
and composites only the cyan expression strokes into the visor before the head
is transformed.  This keeps blink, hit, death, and teleport scanline faces locked
to the head even when the head is tilted or squashed.


## Sheet-locked jump policy

Do not encode the jump arc as per-frame sprite translation.  Keep the `root` stable across all jump frames and let the game object / collision box move vertically at runtime.  Jump animation frames should only change pose: crouch, launch, tuck, descend, and land.  This prevents sprite-cell jitter and keeps the sheet compatible with a runtime movement controller.

## v11 focused arm / hand pass

The default example job is temporarily narrowed to a one-row `run` preview so arm layering and hand placement can be tuned without scanning the full sheet.  Use `examples/robot_rig_job_full.yaml` to render all animations.

Global rig changes made in this pass benefit every animation:

- default arm sprites now use `arm_capsule_vertical` so shoulder-to-wrist anchors form a clean transform axis instead of relying on already-diagonal art;
- raw torso shoulder sockets are nudged down/inward in torso-local space before arms attach, seating arms under the purple shoulder pods;
- hands inherit most of the arm angle via `hand_follow`, so palms/fists stay attached to wrist direction instead of floating independently;
- z-order is now back leg/arm/hand, torso, front leg/arm/hand, head, then front effects.  This keeps arms visible over the body while preventing hands from covering the face in locomotion frames;
- the manifest records arm mount diagnostics (`front_shoulder_raw`, `front_shoulder_target`, `front_wrist`, `hand_follow`, and z-order) for tuning.

## Component-placement debug mode

Use `debug-spritesheet` whenever a pose looks visually jumbled. It renders the same rig and anchors as the production command, but replaces each component with a solid color and draws target anchor markers. Debug sheets intentionally omit text labels by default so the geometry is easier to inspect. This is useful for diagnosing whether a problem is caused by source art noise, bad crop/anchor metadata, bad z-order, or bad pose curves.

```bash
python tools/robot_rig_sheet.py debug-spritesheet examples/robot_rig_job.yaml output/assembled/robot_run_component_debug.png
```

Tune in this order:

1. Check root and hip stability.
2. Check torso/head seating.
3. Check raw shoulder anchors versus corrected shoulder targets.
4. Check arm wrist targets.
5. Check hand wrist anchors after virtual mirroring.
6. Only then tune final art z-order or opacity.

If component blobs overlap in debug mode, fix the pose or anchors before inspecting the production art render.


### v14 side-separated run placement pass

The run row now uses side-separated limb signs and outward/downward shoulder-seat offsets. The front/right limb rotates outward from the right-side shoulder pod, and the back/left limb rotates outward from the left-side shoulder pod, which prevents arms and legs from crossing through the torso. The debug command omits sheet text by default; pass `--keep-sheet-labels` only when labels are useful for screenshots.

## v18 run placement tuning

The run preview remains the focused target while limb placement is tuned.  The
current policy keeps leg components under the torso shell, uses shorter
endpoint-solved arm spans, and scales hands larger relative to the arm capsules
so the character reads as cute instead of skeletal.  Use the no-text debug sheet
when checking whether component anchors remain tied together.

## v23 anchor-first run fix notes

The run pose uses side-specific torso anchors rather than compensating offsets:

- `torso_lean_forward.shoulder_left` is the far/back shoulder on the visible purple pod.
- `torso_lean_forward.shoulder_right` is the near/front shoulder on the forward body side.
- `torso_lean_forward.hip_left` and `hip_right` are tucked under the lower body shell.
- Bent-leg pivots are now tied to the top black connector cap centers.
- Run arm placement is endpoint-solved: shoulder and wrist anchors are tied exactly, and hands snap to the solved wrist anchors.
- Debug head silhouettes are semi-transparent so shoulder/neck attachments remain visible.

Useful debug commands:

```bash
python tools/robot_rig_sheet.py component-anchors examples/robot_rig_job.yaml output/assembled/robot_component_anchors.png --cell 160 torso_front torso_lean_forward arm_capsule_vertical hand_fist hand_fist@flip_x hand_open_down leg_straight_right leg_bent_right leg_straight_left leg_bent_left head_front
python tools/robot_rig_sheet.py debug-frame examples/robot_rig_job.yaml output/assembled/robot_run_frame0_anchor_attach.png --animation run --frame-index 0 --zoom 5 --pad 30 --background black
python tools/robot_rig_sheet.py debug-spritesheet examples/robot_rig_job.yaml output/assembled/robot_run_component_debug.png
```

## Manual anchor editor

When the colored debug view shows connected components floating or collapsing
into the torso, stop tuning offsets and edit the component-local anchors.  Run:

```bash
python tools/anchor_editor.py metadata/robot_components.refined.yaml \
  --slices output/slices \
  --rough-metadata metadata/robot_components.rough.yaml \
  --zoom 6
```

The editor draws each extracted component at integer zoom with anchor crosshairs.
Click/drag to set the selected anchor.  Use arrow keys for one-pixel nudges.
Keep anchor names hidden while visually placing points unless labels are needed
for disambiguation.

Use these placement rules:

- arm anchors go on the center of the actual black connector cap/cuff, not on the
  crop edge;
- torso shoulders go on visible shoulder sockets/pods, split into far/back and
  near/front semantics for side poses;
- torso hips sit under the lower body shell where legs appear to emerge;
- leg hip anchors are the top black connector cap center;
- foot/ground anchors go on the apparent contact point, not the bottom of the
  rectangular crop;
- hand wrist anchors go on the black cuff where the arm connects.

After saving, regenerate the QA images:

```bash
python tools/robot_rig_sheet.py component-anchors examples/robot_rig_job.yaml output/assembled/robot_component_anchors.png --cell 160 torso_front torso_lean_forward arm_capsule_vertical hand_fist hand_fist@flip_x hand_open_down leg_straight_right leg_bent_right leg_straight_left leg_bent_left head_front
python tools/robot_rig_sheet.py debug-frame examples/robot_rig_job.yaml output/assembled/robot_run_frame0_anchor_attach.png --animation run --frame-index 0 --zoom 5 --pad 30 --background black
python tools/robot_rig_sheet.py spritesheet examples/robot_rig_job.yaml output/assembled/robot_run_arm_preview.png
```
